# AIログサマリー仕様設計

本システムの中核である「AIを用いた障害解決の補助（サマリ生成）」に関する機能要件とデータ構造の設計です。

## 1. コア機能

`AI Service` は、Argo WorkflowからのgRPC呼び出しによって起動し、自律的にデータを収集・推論・サマリー生成を行います。

### ステップ 1: データ収集（コンテキストの構築）
引数である `alertData` のメタデータ（発生元Pod名、ネームスペース、時刻範囲など）を基に、以下の情報を自ら収集します。
* 対象時間帯の生ログ（LokiへのLogQLクエリ発行）
* 対象コンテナのリソース状態（PrometheusへのPromQLクエリ発行）
* k8sリソースイベント（任意）

### ステップ 2: LLMによる推論
収集したデータセットと定義済みのプロンプトテンプレートを結合し、LLMに問い合わせます。
期待する回答のJSON構造やマークダウンフォーマットをプロンプトで厳密に指定します。

### ステップ 3: サマリー情報の正規化
LLMからの結果を後続のサービスで扱いやすくするため、以下のような構造化データに変換します。

```json
{
  "title": "[障害検知] frontend-serviceの接続エラー高騰",
  "severity": "HIGH",
  "executive_summary": "DBコネクションプールの枯渇によりサーバーが500エラーを返却しています。",
  "root_cause": "データベースの高負荷によりクエリが遅延し...",
  "suggested_actions": [
    "DBのコネクションプールの最大値を引き上げる",
    "重いクエリ（/metrics等のエンドポイント）の制限",
    "レプリカの追加"
  ],
  "raw_logs_snippet": "..."
}
```

## 2. 画像化・拡張機能に向けた仕様

将来的にテキストだけでなく、「AIによる図解グラフィックの生成」や「メトリクスグラフの埋め込み」などの要件を満たすための設計方針です。

* **マルチモーダル形式**:
  テキストのサマリに加え、Base64エンコードされた画像や、mermaid.js等によるグラフ定義データをJSON内に含められるように拡張フィールド (`attachments`) を用意します。
* **Grafana互換画像表現**:
  Elasticsearchに画像バイナリをそのまま入れるとインデックスが肥大化するため、「Prometheusの当該パネルURLの自動生成リンク」や、「AIが作成した解決策フロー図のMermaid.js文字列」として保持し、Grafana側でレンダリングさせる方式を推奨します。
